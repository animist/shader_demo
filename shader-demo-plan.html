1構造の決定 統一フローフィールド型 (2.5D)

「任せたい」とのことなので決めました。一枚の curl noise 流れ場を“真実の源”にするのが、狙い（一貫したうねり）に対して最も美しく、かつモバイルで現実的です。

domain-warped FBM→ curl noise 流れ場→ 膜の高さ・法線 インクの濃度移流 粒子の移流
  • 3Dに見せるが、フルのレイマーチングはしない。 高さ場→疑似法線→ライティングで screen-space に 2.5D として描く。立体感は出るが負荷は桁で軽い
  • 一貫性 = 美しさ。 面のうねり・インクの拡散・粒子の流れが全部同じ場から生まれるので、見た目に必然性が宿る
  • 安全弁としてレイヤー的退避を内蔵。 重ければ「粒子層だけ低解像度の別パスに逃がす」。＝統一型の美しさ＋レイヤー型の最適化自由度のいいとこ取り

2実装レイヤー構成

流れ場（Flow field）
domain-warped FBM から curl noise を生成。これが全レイヤーの入力。ジャイロの傾きベクトルをここに重力／方向として注入
膜／インク描画パス（2.5D）
流れ場で高さと濃度を作り、法線を起こして疑似ライティング。インクの滲み・縁の発色（玉虫）はここ。フルスクリーン fragment 中心
パーティクルパス（GPGPU）
流れ場に沿って粒子を移流（ping-pong FBO）。加算合成。低解像度で描いて拡大＝安全弁レイヤー
ポスト処理
控えめな bloom／粒状ノイズ／ビネット。質感の最後の仕上げ。負荷を見て増減

3技術スタックと土台

API
WebGL2 ベース
highp 必須化で precision 問題が大幅に軽減。WebGPU(WGSL) は対応端末向けの将来オプション
ライブラリ
OGL（軽量）推奨 / Three.js（潤沢）
フルスクリーン1枚＋粒子なら OGL が軽い。エコシステム重視なら Three.js
最初から組み込むモバイル対策
  • 解像度スケーリング：小さいバックバッファ＋CSS拡大、devicePixelRatio を ~1.5 に上限クランプ
  • precision ガード：#ifdef GL_FRAGMENT_PRECISION_HIGH で highp→mediump フォールバック（WebGL1保険）。実機テスト前提
  • FPS 上限：30〜可変。発熱・バッテリー対策。document.hidden で一時停止
  • 粒子数を端末ティアで分岐、サーマルスロットリング前提で“ピークに頼らない”設計

▶提案する次の一手

最小スキャフォールドを1本作って実機で測る。 フルスクリーンの流れ場＋ジャイロ＋解像度スケーリング＋precision ガードだけを入れた“土台”を先に動かし、iPhone/Android で fps と発熱を確認 → そこから膜・粒子・ポストを盛っていく。盛る前に器の性能を知るのが、想像と実装を最短で近づける道です。

この器が動けば、あとは「同じ流れ場に何を足すか」を実機を見ながら一つずつ判断できます。
